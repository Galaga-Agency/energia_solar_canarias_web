# I18N.md — Internationalization (next-intl)

Skip this file entirely if the project has no i18n.

---

## Install

```bash
npm install next-intl
```

```ts
// next.config.ts
import createNextIntlPlugin from 'next-intl/plugin'
const withNextIntl = createNextIntlPlugin('./i18n/request.ts')
export default withNextIntl(nextConfig)
```

## File Structure

```
locales/               ← ALWAYS "locales/", NEVER "messages/"
  es/common.json, home.json, [page].json
  en/common.json, home.json, [page].json

i18n/routing.ts        ← defineRouting({ locales, defaultLocale })
i18n/request.ts        ← merges locale files, single source of truth

config/i18n.config.ts  ← Language type, routeTranslations
proxy.ts               ← routing proxy (NOT middleware.ts)
```

## i18n/request.ts — Namespace Convention

```ts
// common.json spreads at root. Each page file namespaces under its key.
return {
  locale,
  messages: {
    ...common.default,      // nav, footer, actions, errors — at root
    home: home.default,
    servicios: servicios.default,
    // add one entry per page
  },
}
```

When adding a page: add locale files AND add the import here.

## Component Pattern — Every Component Owns Its Translations

```tsx
// ✅ Each section calls useTranslations directly
import { useTranslations } from 'next-intl'
export function HomeHero() {
  const t = useTranslations('home.hero')
  return <h1>{t('title')}</h1>
}

// ✅ Page controller renders sections with ZERO content props
export function HomeClient() {
  usePageReady()
  return <><HomeHero /><HomeSoluciones /><HomeStats /></>
}

// ❌ Never pass translation strings as props to section components
<HomeHero title={t('hero.title')} />   // BANNED
```

Exception: truly shared layout primitives (FAQAccordion, etc.) may receive translated strings as props, since they can't know their namespace.

## Namespace Pattern

```
useTranslations('nav')             ← common.json nav key
useTranslations('footer')          ← common.json footer key
useTranslations('home.hero')       ← home.json hero key
useTranslations('servicios.faq')   ← servicios.json faq key
```

Always: `useTranslations('[page].[section]')`.

## Constants vs Locales Architecture

**constants/*.ts** — keys, hrefs, structure, pure data. Never translated text.
**locales/**/*.json** — translated strings only.

```
Nav links: constants hold { labelKey, href }, locale JSON holds string for labelKey
FAQs: constants hold key list, locale JSON holds question/answer text
Phone numbers, emails, coordinates → constants, not locales
```

## TranslationContext.tsx — Language Switching Only

```
// Does NOT implement t() or load messages — that's next-intl's job
// Provides: language, changeLanguage(lang), toggleLanguage()
// language = useLocale() from next-intl — the single source of truth.
//   NEVER derive the current locale by parsing usePathname() — it can resolve
//   inconsistently during hydration and cause a hydration mismatch.
// Persists to localStorage "preferred-language"
```

### changeLanguage() — MUST actually navigate (two hard-won rules)

```ts
const router = useRouter()              // next/navigation
const current = useLocale() as Language // next-intl

function changeLanguage(lang: Language) {
  if (lang === current) return
  const nextPath = pathForLocale(pathname, lang)   // default locale → unprefixed "/"
  localStorage.setItem("preferred-language", lang)
  // 1. Set next-intl's NEXT_LOCALE cookie BEFORE navigating.
  document.cookie = `NEXT_LOCALE=${lang};path=/;max-age=31536000;samesite=lax`
  // 2. Navigate with the router (NOT replaceState).
  router.push(nextPath)
  router.refresh()
}
```

**🚫 NEVER use `window.history.replaceState()` to switch language.** It swaps the URL
bar but does NOT make next-intl re-resolve messages — the language silently never changes.
Use `router.push()`.

**🍪 ALWAYS set the `NEXT_LOCALE` cookie before navigating** (with `localePrefix: "as-needed"`).
The proxy's next-intl middleware reads that cookie. If you push to the **default locale's
unprefixed path** (`/`) while the cookie still holds the previous locale, the middleware
**redirects `/` straight back** to the old language — so switching TO the default locale
appears to do nothing. Setting the cookie first makes the middleware agree. (Prefixed
locales like `/fr` happen to work without it because the explicit prefix overrides the
cookie — which is why this bug looks like "only the default locale won't switch".)

## Physical Folder Structure

Routes use CANONICAL default-language slugs only. Never create physical folders for translated slugs.

```
app/[locale]/servicios/page.tsx   ← canonical slug (default language)
// ❌ app/[locale]/services/page.tsx — translated slug handled by proxy.ts, not folders
```

---

## proxy.ts — Routing Proxy

Replaces `middleware.ts` entirely. At project root. Handles:
- Locale detection from URL prefix
- Translated slug → canonical internal path rewriting
- Sets `x-next-intl-locale` header on every response via `withLocaleHeader()`

Every `NextResponse.rewrite()`, `redirect()`, and `next()` must be wrapped with `withLocaleHeader(response, locale)`.
